hello_ffmpeg

通过初识ffmpeg中已经得到了如下的文件了

下面就协议helloffmpeg的程序调用所生成libffmpeg.so的方法

大体思路：

需编译一个so文件，这个so文件是jni方法，可以由java层调用，而这些jni方法里用到的函数就是libffmpeg.so

具体操作如下:

  1. 将生成的libffmpeg.so复制到ndk目录下的platforms/android-5/arch-arm/usr/lib下，因为调用ndk-build的时候需要用到
    eg:

  2. 新建项目，并创建jni目录，在jni目录先再创建ffmpeg目录存放生成的.a,和.h头文件

  1. 接着就是编写jni层调用，

ffmpegUtils_FFmpegPoxy.h文件

ffmpegUtils_FFmpegPorx.hd的实现类:FFmpegPoxy.c,在这里引用了ffmpeg库的avcodec.h的avcodec_version()方法，查看版本号

###接着最关键的一步：编写android.mk文件

android.mk 方式一

LOCAL_PATH := $(call my-dir)  //设置为当前变量

# FFmpeg library
include $(CLEAR_VARS)    //清空变量
PATH_TO_FFMPEG_SOURCE:=$(LOCAL_PATH)/ffmpeg  //定义ffmepg源码的路径
LOCAL_C_INCLUDES += $(PATH_TO_FFMPEG_SOURCE)/include //ffmpeg头文件的路径
LOCAL_LDLIBS := -lffmpeg 这里就是依赖刚才第一步放到lib下的libffmpeg.so
LOCAL_MODULE    := ffmpegUtils_FFmpegPoxy
LOCAL_SRC_FILES := FFmpegPoxy.c
include $(BUILD_SHARED_LIBRARY)

android.mk 方式二（这种方式更好）

LOCAL_PATH := $(call my-dir) :makegun,也就是ndk编译系统提供的方法，获取当前路径

include $(CLEAR_VARS)：CLEAR_VARS由编译i系统提供，指定让GNU MAKEFILE清除除了LOCALPATH变量外的许多LOCAL*变量（例如：LOCAL_MODULE、LOCAL_SRC_FILES等）。这是非常有必要的，因为所有的编译文件都在同一个GUN MKAE执行环境中，所有的变量都是全局变量，不清除容易引起解析错误

PATH_TO_FFMPEG_SOURCE:=$(LOCAL_PATH)/ffmpeg //定义ffmepg源码的路径

LOCAL_C_INCLUDES += $(PATH_TO_FFMPEG_SOURCE)/include //ffmpeg头文件的路径

LOCAL_LDLIBS := -lffmpeg
这行很重要，这是表示你这个so运行的时候依赖于libffmpeg.so这个库， 再举个例子：如果你要编译的so不仅要用到libffmpeg.so这个库还要用的libopencv.so这个库的话，你这个参数就应该写成
LOCAL_LDLIBS := -lffmpeg -lopencv

LOCAL_MODULE := ffmpegUtils_FFmpegPoxy：LOCAL_MODULE变量必须定义，用来标识在Android.mk文件描述的每一个模块。而且名称必须是唯一的，并且不能包含空格。编译系统会自动产生合适的前缀和后缀，比如一个被命名为ffmpegUtils_FFmpegPoxy的共享库模块，将会生成ffmpegUtils_FFmpegPoxy.so文件。

LOCAL_SRC_FILES := FFmpegPoxy.c：模块中的源代码文件，也就是ffmpegUtils_FFmpegPoxy的实现

include $(BUILD_SHARED_LIBRARY)：BUILD_SHARED_LIBRARY是编译系统提供的变量，指向一个GNU Makefile脚本（应该就是build/core目录下的shared_library.mk）,负责收集自从上次调用include $(CLEARVARS)以来，定义在LOCAL*变量中的所有信息，并且决定编译什么，如何正确地去做，并根据其规则生成静态库
